## Findings

### #1 — HIGH (9/10) — Next.js below patched version

- **File:** `apps/web/package.json` declares `"next": "^15.1.0"`
- **Vulnerable range:** `>=13.0.0 <15.5.15` per pnpm audit
- **Why it matters:** Multiple advisories in this range cover middleware/SSR auth bypass classes. Since `clerkMiddleware()` is the only thing standing between an attacker and user financial data, this is load-bearing.
- **Fix:** Bump to `"next": "^15.5.15"`. Run `pnpm install`, `pnpm build`, `pnpm test`, `pnpm test:e2e`. Patch-level only — should be safe.

### #2 — HIGH (8/10) — Build-time RCE chain via PWA dependencies

- **Chain:** `@ducanh2912/next-pwa@10.2.9 → workbox-build → @rollup/plugin-terser → serialize-javascript@6.0.2`
- **Advisory:** GHSA-5c6j-r48x-rmvq (RCE via `RegExp.flags`/`Date.prototype.toISOString`)
- **Why it matters:** Triggered during PWA service-worker bundling. CI runs build with `CLERK_SECRET_KEY`, `DIRECT_URL`, `TURBO_TOKEN` in env — RCE here = secret exfiltration.
- **Fix:** Add `pnpm.overrides` in root `package.json`:
  ```json
  "pnpm": {
    "overrides": {
      "serialize-javascript": ">=7.0.3",
      "effect": ">=3.20.0",
      "flatted": ">=3.4.2"
    }
  }
  ```
  Re-run `pnpm install`, verify `pnpm build` succeeds.

### #3 — MEDIUM (9/10) — Open proxy on stock/FX routes

- **Files:** `apps/web/app/api/stocks/{price,dividend,crypto,tw,us}/route.ts`, `apps/web/app/api/exchange-rate/route.ts`, `apps/web/app/api/cathaylife-rates/route.ts`
- **Why it matters:** No auth, no rate limit. Anyone on the internet can use our deployment to scrape Yahoo Finance / Coingecko. Not classic SSRF (host is fixed) but is an open proxy. Cost amplification on Vercel + IP-ban risk from upstreams.
- **Fix (pick one):**
  1. **Easiest** — add `Cache-Control: public, s-maxage=300, stale-while-revalidate=60` to each route's response. Most requests collapse at Vercel's edge.
  2. **Safer** — extend `apps/web/middleware.ts` matcher to require Clerk session on `/api/stocks/*` and `/api/exchange-rate`. The web client always has a session, so this is invisible to legit users.
  3. **Strongest** — add `@upstash/ratelimit` keyed by IP for these public routes (free tier on Upstash).

### #4 — MEDIUM (8/10) — No rate limiting on user-data routes

- **Files:** `apps/web/app/api/{loans,transactions,entries,portfolio}/**`
- **Why it matters:** `.env.example` declares `RATE_LIMIT_MAX=100` and `RATE_LIMIT_WINDOW_MS=60000` but those env vars are never read. Stolen Clerk session → unbounded `POST /api/transactions` floods the DB.
- **Fix:** Add a per-`userId` token-bucket rate limit in `apps/web/middleware.ts` for `/api/(loans|transactions|entries|portfolio)/.*` using `@upstash/ratelimit` + Vercel KV. Wire to the existing env vars.

### #5 — MEDIUM (8/10) — GitHub Actions are tag-pinned, not SHA-pinned

- **File:** `.github/workflows/ci.yml`
- **Actions:** `actions/checkout@v4`, `actions/setup-node@v4`, `actions/cache@v4`, `pnpm/action-setup@v4`
- **Why it matters:** Tag pins re-resolve every run. If `pnpm/action-setup` maintainer's account is compromised and they re-tag `v4`, the next CI run executes attacker code with `CLERK_SECRET_KEY`, `DIRECT_URL`, `TURBO_TOKEN` in scope.
- **Fix:** Pin every `uses:` to a full commit SHA. Renovate config snippet:
  ```json
  { "pinDigests": true }
  ```
  Or one-off via `gh action-pin`. Document in CONTRIBUTING.md.

## Disclaimer

This tool is not a substitute for a professional security audit. /cso is an AI-assisted scan that catches common vulnerability patterns — it is not comprehensive, not guaranteed, and not a replacement for hiring a qualified security firm. LLMs can miss subtle vulnerabilities, misunderstand complex auth flows, and produce false negatives. For production systems handling financial data and PII, engage a professional penetration testing firm. Use /cso as a first pass to catch low-hanging fruit and improve security posture between professional audits — not as your only line of defense.
